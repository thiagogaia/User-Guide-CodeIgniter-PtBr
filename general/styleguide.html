Guia de Estilo PHP¶

A página a seguir descreve os estilos de codificação adotados ao contribuir para o desenvolvimento do CodeIgniter. Não é necessário usar esses estilos em seu próprio aplicativo CodeIgniter, embora eles sejam recomendados.

Formato do arquivo¶

Os arquivos devem ser salvos com a codificação Unicode (UTF-8).The BOM não deve ser usado. UTF-16 e UTF-32, não há ordem de bytes para indicar em um arquivo codificado UTF-8, e a lista de materiais pode ter um efeito colateral negativo no PHP de envio de saída, impedindo o aplicativo de poder definir seus próprios cabeçalhos. As finalizações de linha Unix devem ser usadas (LF).

Veja como aplicar essas configurações em alguns dos editores de texto mais comuns. As instruções para o editor de texto podem variar; Verifique a documentação do seu editor de texto.

TextMate¶

  1. Abra as Preferências da Aplicação
  2. Clique em Avançado e depois na guia "Salvar"
  3. Em "Codificação de Arquivo", selecione "UTF-8 (recomendado)"
  4. Em "Line Endings", selecione "LF (recomendado)"
  5. Opcional: Marque "Usar para arquivos existentes também" se desejar modificar as finalizações de linha dos arquivos que você abre para sua nova preferência.

BBEdit¶

  1. Abra as Preferências da Aplicação
  2. Selecione "Codificações de texto" à esquerda.
  3. Em "Codificação de texto padrão para novos documentos", selecione "Unicode (UTF-8, sem BOM)"
  4. Opcional: Em "Se a codificação do arquivo não pode ser adivinhada, use", selecione "Unicode (UTF-8, sem BOM)"
  5. Selecione "Arquivos de texto" à esquerda.
  6. Em "Quebras de linha padrão", selecione "Mac OS X e Unix (LF)"

Tag de Fechamento do PHP¶

A tag de fechamento do PHP em um documento do PHP ?> é opcional para o analisador de PHP. No entanto, se usado, qualquer espaço em branco que segue a marca de fechamento, seja introduzido pelo desenvolvedor, usuário ou aplicativo FTP, pode causar saída indesejada, erros de PHP ou, se os últimos forem excluídos, páginas em branco. Por esse motivo, todos os arquivos PHP DEVEM OMITAR a tag de fechamento do PHP e terminar com uma única linha vazia.

Nomes de arquivos¶

Os arquivos de Classe devem ser nomeados de forma semelhante a Ucfirst, enquanto qualquer outro nome de arquivo (configurações, visualizações, scripts genéricos, etc.) deve estar em minúsculas.

INCORRETO:

somelibrary.php
someLibrary.php
SOMELIBRARY.php
Some_Library.php

Application_config.php
Application_Config.php
applicationConfig.php

CORRETO:

Somelibrary.php
Some_library.php

applicationconfig.php
application_config.php

Além disso, os nomes dos arquivos de classe devem corresponder ao nome da própria classe. Por exemplo, se você tem uma classe chamada Myclass, então seu nome de arquivo deve ser be Myclass.php.

Nomeando Métodos e Classes¶

Nomes de classe sempre devem começar com uma letra maiúscula. Múltiplas palavras devem ser separadas com um sublinhado e não CamelCased.

INCORRETO:

class superclass
class SuperClass

CORRETO:

class Super_class

class Super_class {

        public function __construct()
        {

        }
}

Os métodos de classe devem ser inteiramente minúsculos e nomeados para indicar claramente sua função, de preferência incluindo um verbo. Tente evitar nomes excessivamente longos e detalhados. Múltiplas palavras devem ser separadas com um sublinhado.

INCORRETO:

function fileproperties()// não descritivo e precisa de sublinhado separador
function fileProperties()               // não descritivo e usa CamelCase
function getfileproperties()            // Melhor! Mas ainda faltam separador de sublinhado
function getFileProperties()            // usando CamelCase
function get_the_file_properties_from_the_file()        // muito grande

CORRETO:

function get_file_properties()  // Separador descritivo, de sublinhado e todas as letras minúsculas

Nomes de Variáveis¶

As diretrizes para nomeação variável são muito semelhantes às usadas para métodos de classe. As variáveis ​​devem conter apenas letras minúsculas, usar separadores de sublinhado e ser razoavelmente nomeados para indicar seu propósito e conteúdo. Variáveis ​​muito pequenas e não-verbais só devem ser usadas como for() em loops.

INCORRETO:

$j = 'foo';             // variáveis ​​de letra única só devem ser usadas em for() loops
$Str                    // contém letras maiúsculas
$bufferedText           // usa CamelCasing, e pode ser encurtado sem perder o significado semântico
$groupid                // muitas palavras, precisa de separador de sublinhado
$name_of_last_city_used // muito grande

CORRETO:

for ($j = 0; $j < 10; $j++)
$str
$buffer
$group_id
$last_city

Comentários¶

Em geral, o código deve ser comentado de forma prolífica. Não só ajuda a descrever o fluxo e a intenção do código para programadores menos experientes, mas pode ser inestimável ao retornar ao seu próprio código meses abaixo da linha. Não há um formato exigido para comentários, mas os seguintes são recomendados.

Comentários de estilo DocBlock que precedem as declarações de classe, método e propriedade para que possam ser apanhados por IDEs:

/**
 * Super Class
 *
 * @package     Nome do pacote
 * @subpackage  Sub pacote
 * @category    Categoria
 * @author      Nome do Autor
 * @link        http://example.com
 */
class Super_class {

/**
 * Encodes string for use in XML
 *
 * @param       string  $str    Input string
 * @return      string
 */
function xml_encode($str)

/**
 * Data for class manipulation
 *
 * @var array
 */
public $data = array();

Use comentários de linha única dentro do código, deixando uma linha em branco entre grandes blocos de comentários e código.

// quebra os caracteres com nova linha
$parts = explode("\n", $str);

// Um comentário mais longo que precisa dar maiores detalhes sobre o que é
// Ocorrendo e por que pode usar vários comentários de uma única linha. Tente
// Mantenha a largura razoável, cerca de 70 caracteres é o mais fácil de
// ler. Não hesite em vincular a recursos externos permanentes
// Que podem fornecer maiores detalhes:
//
// http://example.com/information_about_something/in_particular/

$parts = $this->foo($parts);

Constantes¶

As constantes seguem as mesmas diretrizes que as variáveis, exceto que as constantes devem sempre ser totalmente maiúsculas. Use sempre constantes CodeIgniter quando apropriado, ou seja, SLASH, LD, RD, PATH_CACHE, etc.

INCORRETO:

myConstant      // separador de sublinhado faltante e não totalmente em maiúscula
N               // nenhuma constante de letra única
S_C_VER         // não descritivo
$str = str_replace('{foo}', 'bar', $str);       // deve usar constantes LD e RD

CORRETO:

MY_CONSTANT
NEWLINE
SUPER_CLASS_VERSION
$str = str_replace(LD.'foo'.RD, 'bar', $str);

VERDADEIRO, FALSO, and NULO¶

As palavras TRUE, FALSE, e NULL sempre devem ser totalmente maiúsculas

INCORRETO:

if ($foo == true)
$bar = false;
function foo($bar = null)

CORRETO:

if ($foo == TRUE)
$bar = FALSE;
function foo($bar = NULL)

Operadores Lógicos¶

O uso do || “or” O operador de comparação é desencorajado, pois a sua clareza em alguns dispositivos de saída é baixa (como o número 11, por exemplo). && é preferível sobre AND e também é aceitável, e um espaço deve sempre preceder e seguir !.

INCORRETO:

if ($foo || $bar)
if ($foo AND $bar)  // Tudo bem, mas não recomendado para aplicativos de destaque de sintaxe comuns
if (!$foo)
if (! is_array($foo))

CORRETO:

if ($foo OR $bar)
if ($foo && $bar) // recomendado
if ( ! $foo)
if ( ! is_array($foo))

Comparando Valores de Retorno e Casting¶

Algumas funções PHP retornam FALSE em caso de falha, mas também podem ter um valor de retorno válido de "" ou 0, que avaliaria FALSE em comparações soltas. Seja explícito, comparando o tipo de variável ao usar esses valores de retorno em condições condicionais para garantir que o valor de retorno seja realmente o que você espera, e não um valor que tenha uma avaliação equivalente de tipo solto.

Use a mesma rigidez ao retornar e verificar suas próprias variáveis Use === e !== se necessário.

INCORRETO:

// If 'foo' se 'foo' estiver no início da string, strpos retornará um 0,
// resultando nesta avaliação condicional como VERDADEIRA
if (strpos($str, 'foo') == FALSE)

CORRETO:

if (strpos($str, 'foo') === FALSE)

INCORRETO:

function build_string($str = "")
{
        if ($str == "") // Uh-oh! E se FALSE ou o inteiro 0 for passado como um argumento?
        {

        }
}

CORRETO:

function build_string($str = "")
{
        if ($str === "")
        {

        }
}

Veja também informações sobre typecasting, o que pode ser bastante útil. Typecasting tem um efeito ligeiramente diferente que pode ser desejável. Ao lançar uma variável como uma string, por exemplo, as variáveis ​​FULL NULL e booleanas se tornam cadeias vazias, 0 (e outros números) se tornam cadeias de dígitos e o VERDADEIRO booleano se torna "1":

$str = (string) $str; // cast $str as a string

Depurando o Código¶

Não deixe o código de depuração nas suas submissões, mesmo quando comentado. Coisas como var_dump(), print_r(), die()/exit() Não deve ser incluído no seu código, a menos que ele atinja um propósito específico além da depuração.

Espaço em Branco em Arquivos¶

Nenhum espaço em branco pode preceder a tag PHP de abertura ou seguir a tag PHP de fechamento. A saída é armazenada em buffer, então o espaço em branco em seus arquivos pode fazer com que a saída comece antes de CodeIgniter produzir seu conteúdo, levando a erros e uma incapacidade para o CodeIgniter enviar cabeçalhos apropriados.

Compatibilidade¶

O CodeIgniter recomenda PHP 5.6 ou mais recente para ser usado, mas deve ser compatível com o PHP 5.3.7. Seu código deve ser compatível com esse requisito, fornecer um retorno adequado ou ser um recurso opcional que falece silenciosamente sem afetar a aplicação de um usuário.

Além disso, não use funções PHP que exijam bibliotecas não padrão para serem instaladas, a menos que seu código contenha um método alternativo quando a função não estiver disponível.

Um Arquivo por Classe¶

Use arquivos separados para cada classe, a menos que as classes estejam intimamente relacionadas. Um exemplo de um arquivo CodeIgniter que contém várias classes é o arquivo da biblioteca Xmlrpc.

Espaço em Branco¶

Use tabs para espaços em branco no seu código, não em espaços. Isso pode parecer uma coisa pequena, mas o uso de guias em vez de espaços em branco permite que o desenvolvedor analise seu código para ter indentação em níveis que eles preferem e personalizam em qualquer aplicativo que eles usem. E como um benefício secundário, resulta em (ligeiramente) arquivos mais compactos, armazenando um caractere de tabulação versus, digamos, quatro caracteres espaciais.

Quebras de Linha¶

Os arquivos devem ser salvos com intervalos de linha Unix. Isso é mais um problema para os desenvolvedores que trabalham no Windows, mas, em qualquer caso, assegure-se de que seu editor de texto esteja configurado para salvar arquivos com quebras de linha Unix.

Indentação do Código¶

Use a indentação estilo Allman. Com exceção das declarações da Classe, as chaves são sempre colocadas em uma linha por si mesmas, e recuadas no mesmo nível que a declaração de controle que as "possui".

INCORRETO:

function foo($bar) {
        // ...
}

foreach ($arr as $key => $val) {
        // ...
}

if ($foo == $bar) {
        // ...
} else {
        // ...
}

for ($i = 0; $i < 10; $i++)
        {
        for ($j = 0; $j < 10; $j++)
                {
                // ...
                }
        }

try {
        // ...
}
catch() {
        // ...
}

CORRETO:

function foo($bar)
{
        // ...
}

foreach ($arr as $key => $val)
{
        // ...
}

if ($foo == $bar)
{
        // ...
}
else
{
        // ...
}

for ($i = 0; $i < 10; $i++)
{
        for ($j = 0; $j < 10; $j++)
        {
                // ...
        }
}

try
{
        // ...
}
catch()
{
        // ...
}

Espaço Entre Chaves e Parênteses¶

Em geral, parênteses e suportes não devem usar espaços adicionais. A exceção é que um espaço deve sempre seguir as estruturas de controle do PHP que aceitam argumentos com parênteses (declarar, fazer-se, elseif, for, foreach, if, switch, while), para ajudar a distingui-los das funções e aumentar a legibilidade.

INCORRETO:

$arr[ $foo ] = 'foo';

CORRETO:

$arr[$foo] = 'foo'; // sem espaços em torno de chaves de matriz

INCORRETO:

function foo ( $bar )
{

}

CORRETO:

function foo($bar) // nenhum espaço em parênteses de declarações de função
{

}

INCORRETO:

foreach( $query->result() as $row )

CORRETO:

foreach ($query->result() as $row) // espaço único que segue as estruturas de controle do PHP, mas não no parêntese interior

Textos de Localização¶

As bibliotecas do CodeIgniter devem aproveitar os arquivos de idiomas correspondentes sempre que possível..

INCORRETO:

return "Invalid Selection";

CORRETO:

return $this->lang->line('invalid_selection');

Métodos e Váriaveis Privadas¶

Métodos e variáveis ​​que só são acessados ​​internamente, como funções de utilitário e auxiliares que seus métodos públicos usam para abstração de código, devem ser prefixados com um sublinhado.

public function convert_text()
private function _convert_text()

Erros do PHP¶

O código deve ser executado sem erros e não confiar em avisos e avisos a serem ocultos para atender a esse requisito. Por exemplo, nunca acesse uma variável que você não configurou (como $_POST array keys) sem primeiro verificá-la. isset().

Certifique-se de que o seu ambiente dev tenha um relatório de erros ativado para TODOS os usuários e que display_errors esteja habilitado no ambiente PHP. Você pode verificar esta configuração com:

if (ini_get('display_errors') == 1)
{
        exit "Enabled";
}

Em alguns servidores onde display_errors está desativado e você não tem a capacidade de alterar isso no php.ini, você pode habilitá-lo com:

ini_set('display_errors', 1);

Nota

Definir a configuração display_errors com ini_set() no tempo de execução não é idêntico a habilitado no ambiente PHP. Ou seja, não terá nenhum efeito se o script tiver erros fatais.

Tags Curtas de Abertura¶

Use sempre as tags de abertura de PHP completas, caso um servidor não tenha habilitado short_open_tag.

INCORRETO:

<? echo $foo; ?>

<?=$foo?>

CORRETO:

<?php echo $foo; ?>

Nota

PHP 5.4 sempre irá ter a tag <?= disponível.

Uma Declaração por Linha¶

Nunca combine declarações em uma linha.

INCORRETO:

$foo = 'this'; $bar = 'that'; $bat = str_replace($foo, $bar, $bag);

CORRETO:

$foo = 'this';
$bar = 'that';
$bat = str_replace($foo, $bar, $bag);

Conjunto de Caracteres¶

Sempre use aspas simples, a menos que você precise de variáveis ​​analisadas, e nos casos em que você precisa de variáveis ​​analisadas, use chaves para evitar a detecção de token malicioso. Você também pode usar seqüências de caracteres duplas, se a seqüência contiver aspas simples, então você não precisa usar caracteres de escape.

INCORRETO:

"My String"                                     // sem variáveis, então não use para aspas duplas
"My string $foo"                                // precisa de chaves
'SELECT foo FROM bar WHERE baz = \'bag\''       // não recomendado

CORRETO:

'My String'
"My string {$foo}"
"SELECT foo FROM bar WHERE baz = 'bag'"

Consultas SQL¶

As palavras-chave SQL sempre são maiúsculas: SELECT, INSERT, UPDATE, WHERE, AS, JOIN, ON, IN, etc.

Divida consultas longas em várias linhas para legibilidade, de preferência quebrando para cada cláusula.

INCORRETO:

// As palavras-chave são minúsculas e a consulta é longa demais para
// uma única linha (... indica continuação da linha)
$query = $this->db->query("select foo, bar, baz, foofoo, foobar as raboof, foobaz from exp_pre_email_addresses
...where foo != 'oof' and baz != 'zab' order by foobaz limit 5, 100");

CORRETO:

$query = $this->db->query("SELECT foo, bar, baz, foofoo, foobar AS raboof, foobaz
                                FROM exp_pre_email_addresses
                                WHERE foo != 'oof'
                                AND baz != 'zab'
                                ORDER BY foobaz
                                LIMIT 5, 100");
